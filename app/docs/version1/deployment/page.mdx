# Deployment

This guide covers deployment options for RegistryAccord services, from local development to production environments.

## Deployment Options

### Local Development

For local development, use the [Devstack](/devstack) which provides a one-command environment with all services and dependencies.

### Standalone Services

Each service can be deployed independently:

- [Identity Service](/services/identity#quick-start)
- [Creator Data Vault](/services/cdv#quick-start)
- [Gateway Service](/services/gateway#quick-start)

### Containerized Deployment

All services are available as Docker images and can be deployed using container orchestration platforms.

## Environment Configuration

### Required Environment Variables

Each service requires specific environment variables for proper operation:

#### Identity Service

- `ID_JWT_SIGNING_KEY`: Base64-encoded Ed25519 private key for JWT signing
- `ID_ISSUER_URL`: Service issuer URL
- `ID_ALLOWED_AUDIENCES`: Comma-separated list of allowed audiences

#### CDV Service

- `CDV_DB_DSN`: PostgreSQL connection string
- `CDV_NATS_URL`: NATS server URL
- `CDV_S3_ENDPOINT`: S3-compatible storage endpoint
- `CDV_S3_BUCKET`: S3 bucket name
- `CDV_S3_ACCESS_KEY`: S3 access key
- `CDV_S3_SECRET_KEY`: S3 secret key
- `CDV_JWT_ISSUER`: Expected JWT issuer
- `CDV_JWT_AUDIENCE`: Expected JWT audience
- `IDENTITY_URL`: Identity service URL for DID validation

#### Gateway Service

- `GATEWAY_JWT_ISSUER`: Identity issuer URL
- `GATEWAY_CDV_BASE_URL`: Base URL for CDV HTTP API
- `GATEWAY_DB_DSN`: Postgres DSN for persisted index (optional)
- `GATEWAY_NATS_URL`: NATS URL for event consumption (optional)

### Configuration Precedence

Configuration follows this precedence:

1. Process environment variables (highest precedence)
2. Environment files (`.env`)
3. Default values (lowest precedence)

Never commit real secrets to version control. Use environment files only for local development.

## Storage Configuration

### PostgreSQL

All services can use PostgreSQL for persistent storage:

- Use connection pooling for better performance
- Configure appropriate connection limits
- Set up proper authentication and access controls
- Regular backups and maintenance

Example connection string:
```
postgresql://user:password@host:port/database?sslmode=require
```

### S3-Compatible Storage

CDV uses S3-compatible storage for media files:

- Configure appropriate access credentials
- Set up proper bucket policies
- Consider using presigned URLs for direct uploads
- Monitor storage usage and costs

### NATS JetStream

For event streaming between services:

- Configure appropriate NATS server settings
- Set up durable consumers for reliable event processing
- Monitor stream and consumer health
- Configure appropriate retention policies

## Security Considerations

### Network Security

- Use TLS for all service-to-service communication
- Restrict network access with firewalls
- Use private networks for internal communication
- Implement proper CORS policies

### Authentication

- Generate secure JWT signing keys
- Rotate keys according to security policies
- Use strong authentication for database connections
- Implement proper access controls

### Data Protection

- Encrypt data at rest
- Use secure storage for sensitive configuration
- Implement proper backup and recovery procedures
- Regularly audit access logs

## Scaling

### Horizontal Scaling

Most services are designed to be stateless and can be scaled horizontally:

- Use load balancers to distribute traffic
- Configure session affinity if needed
- Scale based on metrics and performance
- Implement proper health checks

### Database Scaling

For database scaling:

- Use connection pooling
- Implement read replicas where appropriate
- Optimize queries and indexes
- Consider sharding for large datasets

### Caching

Implement caching strategies:

- Use CDN for static assets
- Implement application-level caching
- Cache JWKS with appropriate TTL
- Use Redis or similar for distributed caching

## Monitoring and Observability

### Health Checks

All services expose health endpoints:

- Liveness: `/healthz`
- Readiness: `/readyz`

Configure your orchestration platform to use these endpoints.

### Metrics

Services expose Prometheus metrics at `/metrics`:

- Monitor request latency and error rates
- Track resource utilization
- Set up appropriate alerts
- Use dashboards for visualization

### Logging

Services use structured JSON logging:

- Centralize logs for easier analysis
- Use correlation IDs for request tracing
- Set appropriate log levels for production
- Implement log retention policies

## Backup and Recovery

### Data Backup

- Regular database backups
- Backup configuration files
- Store backups in secure, geographically distributed locations
- Test backup restoration procedures

### Disaster Recovery

- Document recovery procedures
- Regularly test disaster recovery plans
- Implement failover mechanisms
- Monitor system health proactively

## Example Production Deployment

Here's an example of deploying RegistryAccord services in a production environment:

```yaml
# docker-compose.yml
version: '3.8'
services:
  identity:
    image: registryaccord/identityd:latest
    ports:
      - "8081:8081"
    environment:
      - ID_JWT_SIGNING_KEY=${ID_JWT_SIGNING_KEY}
      - ID_ISSUER_URL=https://identity.example.com
      - ID_ALLOWED_AUDIENCES=cdv,gateway
      - ID_DB_DSN=postgresql://user:pass@postgres:5432/identity
      - PORT=8081
    depends_on:
      - postgres
    restart: unless-stopped

  cdv:
    image: registryaccord/cdvd:latest
    ports:
      - "8082:8082"
    environment:
      - CDV_DB_DSN=postgresql://user:pass@postgres:5432/cdv
      - CDV_NATS_URL=nats://nats:4222
      - CDV_S3_ENDPOINT=https://s3.amazonaws.com
      - CDV_S3_BUCKET=registryaccord-cdv-media
      - CDV_S3_ACCESS_KEY=${CDV_S3_ACCESS_KEY}
      - CDV_S3_SECRET_KEY=${CDV_S3_SECRET_KEY}
      - CDV_JWT_ISSUER=https://identity.example.com
      - CDV_JWT_AUDIENCE=cdv
      - IDENTITY_URL=https://identity.example.com
      - PORT=8082
    depends_on:
      - postgres
      - nats
      - minio
    restart: unless-stopped

  gateway:
    image: registryaccord/gatewayd:latest
    ports:
      - "8083:8083"
    environment:
      - GATEWAY_JWT_ISSUER=https://identity.example.com
      - GATEWAY_CDV_BASE_URL=https://cdv.example.com
      - GATEWAY_DB_DSN=postgresql://user:pass@postgres:5432/gateway
      - GATEWAY_NATS_URL=nats://nats:4222
      - GATEWAY_PORT=8083
    depends_on:
      - postgres
      - nats
    restart: unless-stopped

  postgres:
    image: postgres:15
    environment:
      - POSTGRES_DB=registryaccord
      - POSTGRES_USER=user
      - POSTGRES_PASSWORD=pass
    volumes:
      - postgres_data:/var/lib/postgresql/data
    restart: unless-stopped

  nats:
    image: nats:latest
    command: ["--jetstream"]
    ports:
      - "4222:4222"
    volumes:
      - nats_data:/var/lib/nats
    restart: unless-stopped

volumes:
  postgres_data:
  nats_data:
```

## Kubernetes Deployment

For Kubernetes deployments, create appropriate manifests for each service:

- Deployments for each service
- Services for internal communication
- Ingress for external access
- Secrets for sensitive configuration
- PersistentVolumes for stateful services

Example Kubernetes deployment for Identity service:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: identity
spec:
  replicas: 3
  selector:
    matchLabels:
      app: identity
  template:
    metadata:
      labels:
        app: identity
    spec:
      containers:
      - name: identity
        image: registryaccord/identityd:latest
        ports:
        - containerPort: 8081
        env:
        - name: ID_JWT_SIGNING_KEY
          valueFrom:
            secretKeyRef:
              name: identity-secrets
              key: jwt-signing-key
        - name: ID_ISSUER_URL
          value: "https://identity.example.com"
        - name: ID_ALLOWED_AUDIENCES
          value: "cdv,gateway"
        - name: ID_DB_DSN
          valueFrom:
            secretKeyRef:
              name: identity-secrets
              key: db-dsn
        readinessProbe:
          httpGet:
            path: /readyz
            port: 8081
          initialDelaySeconds: 5
          periodSeconds: 10
        livenessProbe:
          httpGet:
            path: /healthz
            port: 8081
          initialDelaySeconds: 10
          periodSeconds: 30
---
apiVersion: v1
kind: Service
metadata:
  name: identity
spec:
  selector:
    app: identity
  ports:
    - protocol: TCP
      port: 8081
      targetPort: 8081
```

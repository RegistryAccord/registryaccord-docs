# Creator Data Vault (CDV)

The Creator Data Vault (CDV) service provides secure storage and management of creator data with schema validation, JWT authentication, and event streaming. It is the primary write endpoint for creator content in the RegistryAccord protocol.

## Features

- RESTful API endpoints for record and media operations
- Schema validation for supported collections
- JWT/DID authorization
- PostgreSQL storage
- NATS JetStream eventing
- S3-compatible media storage
- Structured logging and observability

## Quick Start

### Requirements

- Go 1.25+
- `make`
- `golangci-lint` (optional for local checks)

### Environment Variables

Key environment variables:

- `CDV_PORT` - HTTP server port (default: 8080)
- `CDV_DB_DSN` - PostgreSQL connection string
- `CDV_NATS_URL` - NATS server URL
- `CDV_S3_ENDPOINT` - S3-compatible storage endpoint
- `CDV_S3_REGION` - S3 region (default: us-east-1)
- `CDV_S3_BUCKET` - S3 bucket name
- `CDV_S3_ACCESS_KEY` - S3 access key
- `CDV_S3_SECRET_KEY` - S3 secret key
- `CDV_JWT_ISSUER` - Expected JWT issuer
- `CDV_JWT_AUDIENCE` - Expected JWT audience
- `IDENTITY_URL` - Identity service URL for DID validation

### Build and Run

```bash
# Build
make build

# Run with environment variables
CDV_DB_DSN="postgres://user:pass@localhost/db" \
CDV_NATS_URL="nats://localhost:4222" \
CDV_S3_ENDPOINT="http://localhost:9000" \
CDV_S3_BUCKET="cdv-media" \
CDV_S3_ACCESS_KEY="minioadmin" \
CDV_S3_SECRET_KEY="minioadmin" \
CDV_JWT_ISSUER="http://localhost:8081" \
CDV_JWT_AUDIENCE="cdv" \
IDENTITY_URL="http://localhost:8081" \
./bin/cdvd
```

## API Endpoints

### Record Operations

#### Create/Update Record

```bash
curl -X POST http://localhost:8082/v1/repo/record \
  -H "Authorization: Bearer YOUR_JWT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "did": "did:plc:abc123",
    "collection": "com.registryaccord.feed.post",
    "record": {
      "text": "Hello, RegistryAccord!",
      "createdAt": "2025-10-13T12:34:56Z"
    }
  }'
```

#### List Records

```bash
curl "http://localhost:8082/v1/repo/listRecords?did=did:plc:abc123&collection=com.registryaccord.feed.post" \
  -H "Authorization: Bearer YOUR_JWT_TOKEN"
```

### Media Operations

#### Initialize Upload

```bash
curl -X POST http://localhost:8082/v1/media/uploadInit \
  -H "Authorization: Bearer YOUR_JWT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "did": "did:plc:abc123",
    "mimeType": "image/jpeg",
    "size": 12345
  }'
```

#### Finalize Upload

```bash
curl -X POST http://localhost:8082/v1/media/finalize \
  -H "Authorization: Bearer YOUR_JWT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "uploadId": "upload-123",
    "did": "did:plc:abc123",
    "sha256": "..."
  }'
```

## Authentication

CDV uses JWT-based authentication:

1. Clients obtain a JWT from the Identity service with `aud=cdv`
2. JWTs are included in requests using the `Authorization: Bearer` header
3. CDV verifies JWTs by:
   - Checking the signature using the Identity service's JWKS
   - Validating the issuer (`iss`) matches `CDV_JWT_ISSUER`
   - Validating the audience (`aud`) includes `cdv`
   - Checking that the token hasn't expired

## Schema Validation

All records are validated against schemas defined in the RegistryAccord specifications:

- Records must conform to their collection's schema
- Schema validation happens at write time
- Invalid records are rejected with detailed error messages
- Schema versions are recorded with each record

## Event Streaming

CDV publishes events to NATS JetStream:

- **Record Events**: Published to `cdv.records.*` subjects
- **Media Events**: Published to `cdv.media.finalized` subject

Event format:
```json
{
  "id": "event-id",
  "type": "com.registryaccord.cdv.record.created",
  "timestamp": "2025-10-13T12:34:56Z",
  "data": {
    // Event-specific data
  }
}
```

## Storage

### PostgreSQL

- Primary structured data storage
- Connection pooling for performance
- Automatic schema migrations
- Support for indexing and querying

### S3-Compatible Storage

- Media files stored in S3-compatible services (MinIO, AWS S3, etc.)
- SHA-256 checksum verification
- Presigned URLs for direct uploads
- Automatic cleanup of temporary files

## Observability

### Health Checks

- Liveness: `GET /healthz`
- Readiness: `GET /readyz`

### Metrics

- Prometheus metrics at `GET /metrics`
- Request latency and count metrics
- Error rate metrics
- Storage backend metrics
- NATS JetStream metrics

### Logging

- Structured JSON logging
- Correlation IDs for request tracing
- Log levels: debug, info, warn, error
- No sensitive data in logs

## Security

- JWT-based authentication with Ed25519 signatures
- Schema validation prevents malformed data
- CORS protection (deny by default)
- Input validation and sanitization
- Secure media upload with checksum verification
- Structured logging without secrets

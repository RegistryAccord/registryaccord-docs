# Services Overview

RegistryAccord is built on a microservices architecture with several specialized services that work together to provide a complete platform for creators. Each service has a specific role and responsibility within the ecosystem.

## Core Services

### Identity Service

The Identity Service manages decentralized identifiers (DIDs) and authentication for the RegistryAccord protocol. It handles:

- DID creation and resolution
- JWT session management via nonce challenge
- Key rotation with overlapping validity windows
- JWKS publication for public key discovery
- Identity recovery mechanisms

[Learn more about the Identity Service](/services/identity)

### Creator Data Vault (CDV)

The Creator Data Vault provides secure storage for creator content and data. It offers:

- Schema-enforced record storage
- Media upload and management
- JWT-based authorization
- Event streaming via NATS JetStream
- S3-compatible storage backend

[Learn more about the CDV Service](/services/cdv)

### Gateway Service

The Gateway service provides read-side APIs for content discovery and social features. It includes:

- Feed generation (following and author feeds)
- Search functionality
- Profile lookups
- Payment stubs for Phase 1
- JWT verification using Identity service JWKS

[Learn more about the Gateway Service](/services/gateway)

## Service Communication

Services communicate through well-defined APIs and event streams:

- **HTTP APIs**: Synchronous communication between services and clients
- **NATS JetStream**: Asynchronous event streaming for real-time updates
- **PostgreSQL**: Shared storage for structured data (when configured)
- **S3/MinIO**: Shared storage for media files

## Authentication Flow

1. Client requests a nonce from the Identity service
2. Client signs the nonce with their private key
3. Client exchanges the signed nonce for a JWT
4. Client uses the JWT to authenticate with other services
5. Services verify JWTs using public keys from Identity's JWKS endpoint

## Data Flow

1. Content is created through the CDV service
2. CDV emits events via NATS JetStream
3. Gateway service consumes events to maintain indexes
4. Clients query the Gateway for feeds and search results
5. Gateway may call CDV directly for certain operations

## Deployment Options

Each service can be deployed:

- **Standalone**: Individually with their own dependencies
- **Devstack**: Together in a local development environment
- **Cloud**: In containerized environments with orchestration

Services are designed to be stateless and horizontally scalable where possible.

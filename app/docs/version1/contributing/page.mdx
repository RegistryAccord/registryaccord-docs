# Contributing

RegistryAccord welcomes contributions from the community! This document outlines how to contribute to the project, including guidelines for different types of contributions.

## Ways to Contribute

There are many ways to contribute to RegistryAccord:

1. **Code contributions** - Bug fixes, feature implementations, improvements
2. **Documentation** - Improving existing docs, writing new guides
3. **Testing** - Running tests, reporting bugs, writing new tests
4. **Feedback** - Using the software and providing feedback
5. **Community support** - Helping other users on forums and issues

## Code of Conduct

All contributors are expected to follow our Code of Conduct, which promotes a welcoming and inclusive environment for everyone.

## Getting Started

1. **Familiarize yourself with the project**
   - Read the documentation
   - Understand the architecture
   - Review the specifications

2. **Set up your development environment**
   - Use the devstack for local development
   - Follow the quickstart guides for each component

3. **Find something to work on**
   - Look for issues labeled "good first issue"
   - Check the roadmap for planned features
   - Propose new features or improvements

## Development Workflow

1. **Fork the repository** you want to contribute to
2. **Create a branch** for your changes
3. **Make your changes** following the coding standards
4. **Write tests** for your changes
5. **Run the test suite** to ensure nothing is broken
6. **Submit a pull request** with a clear description

## Coding Standards

### General Principles

- Keep changes minimal and focused
- Write clear, self-documenting code
- Follow existing patterns in the codebase
- Maintain backward compatibility when possible
- Update documentation when changing public behavior

### Language-Specific Guidelines

#### Go (Services)

- Follow the project's Go coding standards
- Use `gofmt` for formatting
- Run `golangci-lint` for linting
- Write unit tests for new functionality
- Follow the project's error handling patterns

#### TypeScript (CLI)

- Follow the project's TypeScript coding standards
- Use ESLint for linting
- Write unit tests for new functionality
- Follow the project's error handling patterns

### Documentation

- Keep documentation up to date with code changes
- Write clear, concise documentation
- Use examples to illustrate concepts
- Follow the existing documentation style

## Pull Request Process

1. **Before submitting**
   - Ensure your changes pass all tests
   - Update documentation as needed
   - Follow the commit message conventions
   - Ensure your code follows the style guides

2. **Submitting**
   - Create a pull request against the main branch
   - Provide a clear title and description
   - Link to any related issues
   - Request review from appropriate maintainers

3. **During review**
   - Respond to feedback promptly
   - Make requested changes
   - Be open to suggestions
   - Ask questions if anything is unclear

4. **After approval**
   - Squash commits if requested
   - Merge when CI passes
   - Delete your branch after merging

## Reporting Bugs

When reporting bugs, please include:

1. **Clear description** of the issue
2. **Steps to reproduce** the problem
3. **Expected behavior** vs actual behavior
4. **Environment information** (OS, versions, etc.)
5. **Screenshots or logs** if applicable

## Proposing Features

To propose new features:

1. **Check existing issues** to see if it's already been proposed
2. **Create a new issue** with your proposal
3. **Explain the problem** your feature solves
4. **Describe the solution** you have in mind
5. **Consider alternatives** and trade-offs
6. **Be open to feedback** and discussion

## Architecture Decision Records (ADRs)

For significant architectural changes:

1. **Create an ADR** following the template
2. **Explain the context** and problem
3. **Present options** with pros and cons
4. **Recommend a solution** with rationale
5. **Get community feedback** before implementation

## Testing

### Unit Tests

- Write unit tests for all new functionality
- Aim for high code coverage
- Test edge cases and error conditions
- Keep tests fast and isolated

### Integration Tests

- Test interactions between components
- Use the devstack for end-to-end testing
- Test with both in-memory and persistent storage
- Validate API contracts

### Conformance Tests

- Ensure changes pass conformance tests
- Add new tests for new functionality
- Maintain deterministic test behavior

## Documentation Updates

When updating documentation:

1. **Keep it accurate** - Ensure docs match implementation
2. **Keep it clear** - Write for your past self
3. **Keep it complete** - Don't assume prior knowledge
4. **Keep it consistent** - Follow existing patterns
5. **Keep it current** - Update when code changes

## Community

- Join our community channels
- Participate in discussions
- Help other contributors
- Share your experiences
- Be respectful and inclusive
